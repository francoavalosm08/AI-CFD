# Phase 3 Local OpenFOAM No-API Integration Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` if available, or `superpowers:executing-plans` to implement this task-by-task. Use TDD for code changes and verify every task before marking it complete.

**Goal:** Move AI CFD Workbench from fake-mode execution to a deterministic local OpenFOAM runner that does not require a runtime LLM API key.

**Architecture:** React stays on `localhost:5173`, FastAPI stays on `localhost:8000`, and OpenFOAM runs locally through a command adapter. On Windows, the first supported no-Docker target is WSL2 Ubuntu because OpenFOAM is Linux-native. The backend generates OpenFOAM case files from versioned templates, runs OpenFOAM commands step by step, captures logs/artifacts, and shows them in the existing HTML dashboard.

**Tech Stack:** FastAPI, React/Vite, SQLite, PowerShell, WSL2 Ubuntu, Foundation OpenFOAM, Gmsh, Python standard library templates, pytest, Vitest, Playwright.

## Decision Change

The previous Phase 3 plan used Foam-Agent through an MCP server and required `OPENAI_API_KEY`. The new V1 direction is:

- No required runtime API key.
- No required Foam-Agent for V1 real runs.
- No required Docker for V1 real runs.
- Use deterministic OpenFOAM case generation first.
- Use WSL2 Ubuntu as the first local OpenFOAM runtime on Windows.
- Keep the existing Foam-Agent MCP code as an optional future/advanced path until the deterministic runner is proven.

The product should still feel like an AI CFD workbench from the user side: the browser asks for simulation specs, then the backend runs the workflow. The difference is that the V1 solver path is generated by local code and templates instead of by a live LLM call.

## Why This Avoids The API Key

The API key was only needed because Foam-Agent uses an LLM provider to plan/write/review OpenFOAM cases. OpenFOAM itself does not need an API key.

This plan replaces:

```text
FastAPI -> Foam-Agent MCP -> LLM API -> generated OpenFOAM case -> OpenFOAM
```

with:

```text
FastAPI -> deterministic case builder -> local OpenFOAM commands -> artifacts
```

An IDE LLM can still help develop and debug the project, but it is not part of the app's runtime path.

## Decisions Locked

- Keep fake mode as the fast regression path.
- Add a new local runner mode, tentatively `FOAM_AGENT_MODE=local_openfoam` or a clearer replacement such as `CFD_RUNNER_MODE=local_openfoam`.
- Start with incompressible, steady, external aerodynamics.
- Start real acceptance with `.msh` input only.
- Use WSL2 Ubuntu as the first non-Docker OpenFOAM target on Windows.
- Use deterministic templates for case files.
- Preserve the existing API routes and UI workflow.
- Preserve the current Foam-Agent MCP implementation as optional code until the local runner passes acceptance.
- Do not attempt heat transfer, vibration, aeroelasticity, or Ansys in this phase.

## What Could Go Wrong With Changing Direction Now

| Risk | Why It Matters | Mitigation | Verification |
| --- | --- | --- | --- |
| Existing MCP code becomes confusing | The repo already contains Foam-Agent scripts/tests/docs | Mark MCP as optional, keep it passing, and make the no-API path the documented next milestone | `release-check.ps1` still passes; docs point to local runner plan |
| Docs contradict each other | README, AGENTS, runbooks currently mention API keys | Update all handoff docs together | `rg "OPENAI_API_KEY|Foam-Agent|Docker" docs README.md AGENTS.md` review |
| Removing MCP too early breaks tested behavior | MCP tests currently add useful coverage | Do not delete MCP code in the pivot commit | `cd backend; ..\.venv\Scripts\python.exe -m pytest` |
| WSL2 path mapping breaks uploads | Windows paths and Linux paths differ | Add a tested path adapter from Windows repo paths to WSL `/mnt/c/...` paths | unit tests for spaces and OneDrive paths |
| OpenFOAM version mismatch | Different installs may have different utilities | Add a preflight script that checks `wsl`, OpenFOAM env, `gmshToFoam`, `checkMesh`, solver commands | `scripts/dev-openfoam-wsl.ps1 -CheckOnly` |
| Boundary names from uploaded `.msh` are unknown | Deterministic templates need inlet/outlet/wall patches | Add a mesh inspection step and a UI/API error if required patches are missing | tests with sample `gmshToFoam` boundary output |
| Mesh conversion succeeds but case is physically bad | CFD automation can create invalid setups | Add conservative defaults, visible warnings, `checkMesh` gate, and logs | smoke run fails before solver if `checkMesh` fails |
| No visualization appears | OpenFOAM may run but no images are generated | Always capture logs and zip case; add `foamToVTK`/CSV outputs first; PNGs can be second | artifact discovery tests |
| Manual step-by-step control conflicts with automation | The user wants IDE-controlled steps, but the app needs repeatability | Add a dry-run/plan mode that writes commands before executing them | smoke dry-run verifies command manifest |

## Target Runtime

Run two app processes and one OpenFOAM runtime:

1. **Frontend**
   - Command: `.\scripts\dev-frontend.ps1`
   - URL: `http://localhost:5173`
   - Role: upload form, external-aero specification form, run progress, result dashboard.

2. **Backend**
   - Fake command: `.\scripts\dev-backend.ps1`
   - Local OpenFOAM command to add: `.\scripts\dev-openfoam-backend.ps1`
   - URL: `http://localhost:8000`
   - Role: upload storage, deterministic case generation, command orchestration, SQLite metadata, event stream, artifact serving.

3. **OpenFOAM Runtime**
   - First target: WSL2 Ubuntu with OpenFOAM installed.
   - Command checks to add: `.\scripts\dev-openfoam-wsl.ps1 -CheckOnly`
   - Role: execute `gmshToFoam`, `checkMesh`, solver, post-processing, and artifact export.

## New Backend Components

Create these modules instead of expanding the Foam-Agent client:

- `backend/app/openfoam/case_builder.py`
  - Converts `SimulationSpec` plus uploaded mesh metadata into a deterministic OpenFOAM case directory.
  - Writes `0/`, `constant/`, and `system/` dictionaries from templates.
  - Records a `case-manifest.json` with all generated files and commands.

- `backend/app/openfoam/templates.py`
  - Holds template rendering helpers for OpenFOAM dictionaries.
  - Keeps templates versioned in Python strings or separate `.jinja`/`.foam` files.

- `backend/app/openfoam/runner.py`
  - Defines `OpenFoamCommandRunner`.
  - Runs commands through an injected command executor.
  - Supports dry-run and real execution.
  - Emits progress events.

- `backend/app/openfoam/wsl.py`
  - Maps Windows paths to WSL paths.
  - Runs `wsl.exe bash -lc "<command>"`.
  - Sources OpenFOAM environment before commands.

- `backend/app/openfoam/parsers.py`
  - Parses `checkMesh` output, residual logs, force coefficient logs, and command failures.

- `backend/app/openfoam/artifacts.py`
  - Copies logs, CSV files, generated VTK files, and zipped case into the app run directory.

Keep existing `backend/app/foam_agent.py` as optional MCP code. Do not use it in the no-API runner path.

## Environment Variables

No runtime API key is required for the local OpenFOAM path.

Recommended local variables:

```powershell
CFD_RUNNER_MODE=local_openfoam
OPENFOAM_RUNTIME=wsl
OPENFOAM_WSL_DISTRO=Ubuntu
OPENFOAM_BASHRC=/opt/openfoam*/etc/bashrc
OPENFOAM_RUN_TIMEOUT_SECONDS=1200
```

Optional advanced Foam-Agent mode can keep using:

```powershell
OPENAI_API_KEY=sk-...
FOAM_AGENT_MODE=mcp
FOAM_AGENT_URL=http://127.0.0.1:7860/mcp
```

Those optional variables must not be required for default V1 real runs.

## Deterministic External Aero V1 Scope

The first deterministic case type is:

- Solver: `simpleFoam`
- Flow: incompressible, steady RANS
- Turbulence: SST k-omega or a conservative default supported by the installed OpenFOAM version
- Input mesh: `.msh` first
- Mesh import: `gmshToFoam`
- Required boundary concepts: inlet, outlet/farfield, wall/body
- Outputs:
  - solver logs
  - residuals CSV
  - force/force coefficient logs when configured
  - `checkMesh` report
  - zipped OpenFOAM case
  - VTK output if `foamToVTK` is available
  - PNG images later if local PyVista is installed

STEP/STL support stays best-effort and can continue through Gmsh conversion before this runner receives a `.msh`.

## Step-By-Step Command Manifest

Each run should write `openfoam-commands.json` before executing anything:

```json
{
  "runtime": "wsl",
  "case_dir_windows": "C:\\Users\\franc\\OneDrive\\Desktop\\AI CFD\\.local-data\\runs\\run-id\\case",
  "case_dir_wsl": "/mnt/c/Users/franc/OneDrive/Desktop/AI CFD/.local-data/runs/run-id/case",
  "commands": [
    {
      "name": "import_mesh",
      "command": "gmshToFoam input.msh",
      "required": true
    },
    {
      "name": "check_mesh",
      "command": "checkMesh -allGeometry -allTopology",
      "required": true
    },
    {
      "name": "solve",
      "command": "simpleFoam",
      "required": true
    },
    {
      "name": "export_vtk",
      "command": "foamToVTK",
      "required": false
    }
  ]
}
```

This gives the "tell my IDE to do the steps" workflow without making the app depend on a live LLM API.

## Acceptance Criteria

No-API Phase 3 sample acceptance is complete when:

- The default documented real-solver path does not require an API key.
- A `.msh` upload can generate a deterministic OpenFOAM case.
- The backend writes the exact command manifest before executing.
- Dry-run mode works on a machine without OpenFOAM.
- WSL/OpenFOAM preflight is clear and actionable.
- A real WSL/OpenFOAM run can complete or fail with useful logs/artifacts.
- Fake mode still passes `scripts/release-check.ps1`.
- Foam-Agent MCP remains optional and does not block normal V1 usage.

That acceptance slice is complete on this development machine for the committed `samples/external_box.geo` sample and the generated NACA 4412 airfoil validation case. The next milestone is testing more user-provided aircraft/vehicle `.msh` files against the documented physical-name contract.
